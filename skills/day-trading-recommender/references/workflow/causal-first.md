# 日内筛选工作流

本文件承接 `day-trading-recommender` 的候选生成、审计池、用户点名防锚定、强制三连校验与复盘流程。输出格式看 [output-contract.md](../core/output-contract.md),风控看 [risk-rules.md](../core/risk-rules.md),报价认证看 [web-cross-check.md](../core/web-cross-check.md)。

## 固定加载顺序

1. 先读 [output-contract.md](../core/output-contract.md),锁定最终回答结构
2. 再读 [risk-rules.md](../core/risk-rules.md),确认仓位、止损、止盈、用户状态守门和概率分档
3. 再读 [web-cross-check.md](../core/web-cross-check.md),确认报价、新闻、红旗三方认证
4. 再读本文件,执行候选生成、防锚定、三连校验和模式路由
5. 最后按路由读取 1-2 个 mode 文件,只使用模式独有逻辑

如果 MCP 不可用,必须说明本次走 WebSearch 回退路径,不得伪装成完整量化筛选。

## 数据新鲜度前置检查

每次调 skill 第一步跑 `db_health`:

- `yfinance.latest_date` 是今日或昨日 → 正常执行
- `yfinance.latest_date` 距今 > 2 个交易日 → 明确告诉用户"本地数据库未同步,日内判断改为只用 WebSearch + 你提供的盘前报价",不要硬用旧数据
- `earnings_calendar.last_sync` 超 24 小时 → 财报跳空模式需 WebSearch 双验证

## 数据源与 schema

优先顺序:

1. WebSearch / 公开源:`premarket gainers`、`top gainers`、`intraday top gainers`、`unusual volume movers`、`trading halt / LULD resume`、公司 PR/IR、SEC/8-K/offering/ATM/reverse split
2. TradingAgents MCP:`db_health`、`latest_prices`、`symbol_snapshot`、`query_market_data`、`long_momentum_candidates`、`earnings_dates`、`company_fundamentals`

本地日线表固定为 `market_data.us_equity_daily_prices`;日期字段为 `price_date`;默认 `provider='yfinance'`。手写 SQL 或执行 scripts 时不得使用旧表名 `us_equity_daily` 或旧字段 `date`。

## 用户点名防锚定硬门

用户点名的 ticker 只能进入`待验证池`,不能自动进入推荐表,更不能因为用户刚问过就升为主攻。只要用户的请求是"规划资金 / 推荐股票 / 盘前扫描 / 做 T + 财报埋伏"这类组合决策,必须先重新拉一轮全市场候选:

- WebSearch 至少查盘前/盘中 movers、财报后 movers、今日催化/upgrade/buyback/order/partnership
- 本地至少补 `gap_candidates.sql`、`squeeze_candidates.sql`、`micro_momentum.sql`、财报预判池/财报后首日池中的相关候选
- 点名票和全市场候选使用同一评分表、同一流动性门、同一资本事件门、同一 R/R 门
- 点名票只有在综合排序进入 Top 3,且主导偏向强于或不弱于未点名候选时,才可列为`主攻/备攻`
- 若点名票只是"看起来不错",但排名不如全市场候选,必须输出为`观察/不推荐`,并说明它输给了哪些候选、输在哪些指标
- 最终输出必须标注每只票来源:`全市场扫描` / `用户点名复核` / `两者重合`

如果没有完成全市场候选对照,只能回答"单票复核",不得给资金组合推荐。

## 强制日内三连校验

不管哪个模式,**对最终准备首推的候选**先定义一个主导偏向,再按顺序跑以下三项。任何一项红灯若直接推翻主导偏向,则排除或降级:

### Step -1. 当前可交易性硬门

按 [web-cross-check.md](../core/web-cross-check.md) 执行:

- 报价证据等级 A/B 才允许 `主攻/备攻`、入场、仓位、正常概率
- 报价获取顺序固定:Web movers → 新闻催化 → 本地日线
- 用户券商报价优先级最高,给出后必须重算入场/止损/止盈/仓位/R/R
- 偏离日线收盘 ≥5% 时,基于实时/延长价重算
- 拿不到当前价 → `观察/待盘前复核`,不得给"现在买"

### Step 0. 主导偏向先行

每只日内候选必须先归因到一个**单一交易驱动**,只能先选一个主标签:

- `财报预期差`
- `新鲜催化`
- `榜首龙头惯性`
- `低流通 squeeze`
- `事件窗口`
- `突破扩张`

先写一句:

- `这单不是因为它面面俱到,而是因为 ______`

**如果没有明确主导偏向,默认不首推。**

### Step A. 流动性门(硬线)

详见 [risk-rules.md](../core/risk-rules.md) "流动性门"。简表:

- 默认 `20 日均成交额 ≥ $5,000,000`,当日量比 `≥ 1.5x`
- 高弹性例外:`≥ $2,000,000`(标"高波动,滑点更大")
- 财报预判例外:`≥ $10,000,000`(若历史/指引/板块/红旗 4 项 ≥ 3 项)
- 微盘例外:`≥ $500,000`(标"低流动性,仓位受限")

不满足直接排除,不要"可惜了"式挽留。

### Step B. 资本事件门(硬线)

WebSearch `"{ticker} 8-K OR offering OR ATM OR reverse split last 7 days"`:

- 过去 7 天有公告增发/ATM/反向拆股/破产/SEC 调查 → **直接排除**
- 微盘模式额外查:历史 52 周高是现价 10 倍以上 → 有合股历史,标高危,只允许极端彩票仓

详见 [red-flags.md](red-flags.md)。

### Step C. R/R 门(硬线)

- 入场上限价 → 硬止损 → 第一止盈
- **R/R ≥ 1.5** 才保留;否则"今天这只不值得做"
- **ATR 高不是原罪**。只要存在回踩/二次突破结构、止损清晰且第一止盈仍有足够空间,高 ATR 可以成为主攻候选
- 只有在**已接近日内极端高点 / 无回踩空间 / 已第 3 天尾声**时,才把高波动票降级
- 详细计算法则在 [risk-rules.md](../core/risk-rules.md) "止损硬线"

### 解释纪律

- 三连校验不是为了证明它"所有优点都齐了"
- 而是为了确认"这一单最核心的那口肉,没有被流动性/资本事件/RR 直接毁掉"

## 时间段路由

输出时必须先判断**当前美东时间**处于哪个交易阶段,再决定怎么写操作建议:

| 时段 ET | 默认动作 |
|---|---|
| < 09:30 | 盘前:挂条件单、盘前涨跌幅、盘前高低点、给"开盘后 09:45 ET 再确认"的计划 |
| 09:30-09:45 | 开盘观察期:不写"立即追入";优先"等 VWAP / 首个 5 分钟区间确认" |
| 09:45-10:30 | 主要建仓窗口 |
| 10:30-11:30 | 若第一止盈未触发,减仓 30% |
| 11:30-13:30 | 午间低波动,不加仓,观察 |
| 13:30-14:30 | 若动量仍在,持有;否则减仓 |
| 14:30-14:45 | 最后平仓窗口 |
| ≥ 14:45 | 不再给日内新开仓建议;只允许"今天放弃 / 等明天" |

详见 [output-contract.md](../core/output-contract.md) "时间上下文自适应" 与 [risk-rules.md](../core/risk-rules.md) "时间纪律"。

## 模式路由特殊规则

### 财报预判优先级提升

- 当用户明确表达"盈利关键在财报预判 / 想吃第一口肉 / 愿意承担高风险"时,**优先进入 `财报预判埋伏` 模式**,不要先用普通日内模式把财报 `<=7` 天的票一刀切排除
- 即使用户未点名该模式,若请求明显围绕"财报前布局",也应先读取 [modes/earnings-pre-position.md](../modes/earnings-pre-position.md)
- `高赔率进攻雷达` 在默认扫描时,必须把**未来 0-10 天财报窗口候选池**作为并行候选源之一,不能只扫财报后首日
- 用户问"今天出财报 / 今晚开奖 / 今天能赌财报吗",先进入 `财报预判埋伏` 的**当天确认池**流程,不要把 `days_until=0` 误排除

### 事前催化雷达优先级提升

- 当用户明确问"能不能吃第一口 / 事前预判 / 爆发前找票 / 还没涨之前埋伏"时,优先进入 `事前催化雷达`,不要先输出已经大涨的 movers
- `事前催化雷达` 必须把候选分成`可提前埋伏`、`只设提醒`、`等公告后确认`三类
- 若催化没有明确日期或可验证窗口,只能列为观察,不得给重仓配置

### 生科例外

用户提到具体生科 ticker 且其市值 < $500M 时,优先走临床事件窗口模式(默认禁令的窄白名单豁免),而不是 micro-pump。

## 关键判断纪律

1. **用户想抓单次大肉** → 默认优先输出高弹性主攻机会,不要先退回稳健票
2. **用户明确说盈利关键在财报预判** → 优先输出 `财报预判埋伏` 结果,普通日内候选降为补充
3. **用户问"为什么没推到 X"(事后 +200% 的票)** → 诚实说"事前必须区分高赔率扩张和尾声残骸",然后用龙头追击、高赔率雷达或 `财报预判埋伏` 帮他找**尚未失去赔率**的类似候选
4. **所有候选都不过 R/R 1.5 或都已尾声** → 明确说"今天没有值得做的大肉机会,建议空仓或等 10:30 ET 后重新扫描",不硬凑
5. **数据库不新鲜或工具失败** → 不伪造候选。告诉用户"数据不够,请你提供正在看的标的,我做基本面 + 风险评估"
6. **高波动 ≠ 自动彩票仓**。只有纯 pump、无源拉升、资本事件红旗重、或已第 3 天尾声时,才降到极端彩票
7. **默认不要把多个利多硬拼成结论**。日内交易更常见的赚钱来源是某一个驱动突然压过市场,不是面面俱到

## 复盘闭环

用户要求复盘昨日/上次推荐时,用 `latest_prices` 或 `symbol_snapshot` 拉推荐后最新完整交易日的 `open/high/low/close/volume/volume_ratio/dollar_volume`,与原入场、止损、第一止盈、第二止盈对照。

复盘必须记录:

- `命中`:触发入场后是否达到第一/第二止盈,是否符合原主导偏向
- `失败`:是否触发止损、跳空低开、跌破 VWAP/开盘价/承接区,以及原推荐哪里过度乐观
- `遗漏`:当日实际涨幅榜/成交额榜、盘后公告榜、盘前/夜盘榜是否有更强票未进入审计池
- `复用`:成功或失败是否值得写入 [case-studies.md](../case-studies.md),并沉淀成可执行规则

复盘只能强化规则和案例库,不能声称未来收益有保证。
