# 模式 8:财报预判埋伏

**场景**: 财报前 `0-10` 天,希望通过预期差筛出最可能在财报后走出 `+12%` 到 `+50%` 跳空的股票,提前吃第一口肉。  
**持仓**: 财报前建仓,持有到财报后 `0-2` 个交易日。  
**目标**: 不是提高胜率,而是提高**赔率密度**和信息浓缩度。

**定位**:
- 这是 `day-trading-recommender` 里的**高优先级事件模式**
- 它和 `财报跳空首日` 不同: 这里是**财报前下注**
- 它和 `long-stock-recommender` 的 `财报跳空` 共享底层逻辑,但这里更强调**浓缩信息输出**和**可执行性**

## 触发条件

用户关键词:
- "财报预判"
- "提前埋伏财报"
- "赌财报"
- "今天出财报"
- "今天有哪些财报"
- "今晚开奖"
- "吃第一口肉"
- "财报前布局"
- "财报预期差"
- "earnings pre-position"

当用户明确表示"盈利关键在财报预判"时,即使没说这些词,也应优先进入本模式。

## 核心原则

1. **普通日内排除财报 ≤7 天**,本模式反过来专门研究这类票
2. 这不是普通趋势交易,而是**事件驱动下注**
3. 不允许用"看起来挺强"代替历史反应与指引分析
4. 输出要高度浓缩,方便用户快速比较,但信息必须足够支撑决策
5. 用户问"今天/今晚"时,必须先做**当天确认池**,不能把 `days_until=0` 误排除,也不能把未来几天票混进当天推荐

## 筛选流程

### Step 0AA - 用户时点路由(新增硬要求)

先判断用户问的是哪一种财报问题:

- `今天出财报 / 今晚开奖 / 今天能赌财报吗` → **当天确认池**
- `这周 / 近一周 / 未来几天财报` → **近一周预筛池**
- `提前埋伏 / 财报前布局` 且未指定今天 → **3-10 天埋伏池**

执行顺序固定为:

1. 本地数据库先取未来 `0-7` 或 `0-10` 天财报池,并保留 `earnings_date/timing/is_estimated/eps_estimate`
2. 用日线库补 `close / dollar_volume / volume_ratio / 5d/20d return / 20d range / distance from 20d high`
3. 对流动性和波动足够的候选,逐只用官方/IR/Web 二次确认财报日期
4. 若用户问"今天",只保留官方确认 `earnings_date=今天` 且尚未公布的票;BMO 已公布的票移到"结果后反应交易",不得当作财报前埋伏
5. 对最终候选跑历史财报反应、红旗、预期差和交易计划

输出中必须标注:

- `数据库日期`
- `官方确认日期`
- `是否 estimated`
- `确认结果`: `一致 / 官方修正 / 未确认降级 / 已公布移出`

### Step 0A - 财报日期二次确认(新增硬要求)

在输出任何候选前,必须先确认**这只票到底是不是还没出财报**。

执行:
- 本地 `earnings_dates(within_days=10)` 只作为第一层筛选
- 对最终准备输出的每只候选,再 WebSearch:
```text
"{ticker} earnings date investor relations"
"{ticker} reports results {current_month} {current_year}"
"{ticker} q1 q2 results date site:globenewswire.com OR site:businesswire.com OR site:prnewswire.com"
```

规则:
- 若 Web 显示财报已经在昨晚/今晨公布 → **立刻移出本模式**
- 若本地库日期和 Web/IR 不一致 → **以更近期的官方来源为准**
- 若日期不确定或多源冲突 → **降级为不推荐**,因为时机错位比选股错更致命
- **官方 IR / SEC / 公司 PR > MarketBeat / StockAnalysis / TipRanks 等 estimated 页面**

**新增纪律**:
- `days_until ≤ 7` 的票,必须做日期二次确认
- 如果不能确认具体日期,就不能写成 A/B 级候选
- 如果数据库只有 `is_estimated=true`,但官方 IR / 公司公告确认了同一天,可升级为"官方确认";如果只找到 MarketBeat/StockAnalysis/TipRanks estimated 页面,仍保持降级

### Step 0 - 前置风险门

若以下任一成立,直接拒绝本模式:
- 用户明显情绪化、想翻本、刚亏大钱
- 用户只想押 `1` 只且重仓
- 市场系统性环境太差(如 VIX 高企、成长股财报普遍被砸)

### Step 1 - 建立财报窗口池

调 `earnings_dates(within_days=10)` 或 SQL 找未来 `0-10` 天公布财报的股票。若用户问"今天",先取未来一周池,再收敛为今天官方确认池。

优先保留:
- `days_until` 在 `0-7` 天;若做提前埋伏,优先 `3-7` 天;若做当天开奖,允许 `0` 天但必须官方确认且未公布
- 市值 `$0.5B-$30B`
- 20 日均成交额 `>= $10M`
- `RSI14 < 68`
- 当前价距离 20 日高至少 `5%` 以上,避免财报前过热

当天开奖例外:
- 若是高 beta 主题票(量子、AI 算力、BTC 矿股、核能、太空/eVTOL),即使距离 20 日高不足 `5%`,可以保留为彩票候选,但必须降一级并写"已接近高位,只小仓"
- 若财报时间是 BMO 且当前已过公布时点,只能做"结果后反应交易",不输出为埋伏

### Step 2 - 历史财报反应(一票否决)

Read 并执行 [../../../long-stock-recommender/scripts/earnings_reaction_check.sql](../../../long-stock-recommender/scripts/earnings_reaction_check.sql)。

重点记录:
- 过去 `4` 次财报次日上涨次数
- 过去 `4` 次中,是否有"EPS beat 仍跌"
- 过去 `4` 次财报后 `3` 日累计表现
- 最近 `1-2` 次财报后是否出现"盘中先深砸,收盘再拉回"

规则:
- `>=3/4` 次上涨: 🟢
- `2/4` 次上涨: 🟡
- `<=1/4` 次上涨: 🔴 直接排除
- 若最近 `2` 次里出现过明显的`先砸后修复`路径,即使收盘结果不差,也至少降一级

解释:
- 我们不是做回测表格,而是在真钱里埋伏财报
- `收盘勉强收平` 和 `开盘先被砸穿 8% 再拉回` 对持仓体验完全不是一回事
- 像 GCT 这类路径差的票,不能因为最后收回来就继续按高质量预判票处理
- 若公司这次财报已经验证出"beat 了但市场仍然先砸",后续再遇到类似形态时,默认先尊重价格反应,不要迷信文字面超预期

### Step 3 - 一致预期和指引方向

WebSearch:
```text
"{ticker} earnings preview {current_year}"
"{ticker} estimate revisions {current_year}"
"{ticker} guidance raised lowered {current_year}"
"{ticker} q1 q2 demand pricing outlook {current_year}"
```

重点提炼:
- 分析师一致预期最近 30 天是上修还是下修
- 管理层上一季给的指引是否保守
- 所在行业近期是否已有同业先爆超预期

打分:
- 一致预期上修 + 指引可上修: `+2`
- 只有其中一项: `+1`
- 两项都弱: `0`

### Step 4 - 板块景气传导

WebSearch:
```text
"{sector} earnings this week beat guidance"
"{peer1} {peer2} earnings beat raised guidance"
```

如果上游/同业最近财报明显偏强,给板块传导加分。

### Step 5 - 资本事件红旗

必须查:
- offering / ATM / convert / warrants
- reverse split
- insider 大额减持
- 做空报告 / SEC / 会计问题

命中即降级或排除。

### Step 5A - 财报后路径风险(新增)

若 Web/复盘资料显示该票历史常见以下路径之一:

- `财报后先大跌,尾盘才回本`
- `盘前冲高,开盘一路兑现`
- `首小时跌破昨收/盘前低点后才慢慢修复`

则处理:

- A → B
- B → C
- C → 直接放弃

因为本模式赚的是**预判差 + 开盘后立即定价**,不是赌尾盘修复。

## 输出规则: 信息浓缩版

先给总表,默认只保留 `3-5` 只。

### 总表

```md
| 票 | 数据库日期 | 官方确认 | 预判等级 | 历史反应 | 预期差来源 | 入场区间 | 失效位 | 财报后第一目标 | 建议仓位上限 | 一句话 |
|---|---|---|---|---|---|---|---|---|---|---|
| TICKER | MM-DD est/AMC | MM-DD AMC/BMO/未确认 | A/B/C | 3/4涨 | 指引/上修/板块 | $X-$Y | $X | $X | $XXX (X%) | 为什么赌它 |
```

### 单票展开

每只必须回答以下 8 项,不写废话:

```md
候选 N: {TICKER}
模式: 财报预判埋伏
预判等级: A / B / C

关键信息
- 财报时间: {date} {bmo/amc}
- 历史财报反应: 过去 4 次次日 {X/4} 上涨
- 财报日期确认: 数据库日期 / 官方 IR 或公司公告 / 是否一致;若本地为 estimated,说明是否被官方确认
- 预期差来源: 指引可能上修 / 一致预期上修 / 板块传导 / 成本改善
- 路径风险: 历史上是否常见先砸后拉
- 最大红旗: <一句话>

交易计划
- 当前价: $X.XX
- 财报前入场区间: $X.XX-$Y.YY
- 失效位: $X.XX
- 财报后第一目标: $X.XX
- 财报后第二目标: $X.XX
- 建议仓位上限: $XXX (总资金的 X%)
- 计划买入股数: X 股

一句话结论
- <为什么值得埋伏 / 为什么只配小仓 / 为什么放弃>
```

## 仓位纪律

- A 级: 单票 `2.0%-2.5%`
- B 级: 单票 `1.0%-1.5%`
- C 级: 单票 `0.5%-1.0%`
- 总财报预判仓: `<=15%`
- 默认组合化,但如果用户只问单票,仍必须明确写出"单票下注不如组合"

## 什么时候明确放弃

- 财报只剩 `<=2` 天,已经没有舒服建仓窗口
- 用户明确问"今天/今晚"时,`<=2` 天不自动放弃;改按**当天确认池**执行,但仓位必须比普通埋伏更小
- 财报日期本地库与 Web 不一致,且无法确认哪个最新
- RSI 已经 `>75`
- 最近 30 天有融资/反拆/毒融资
- 历史财报反应太差
- 历史路径经常是先砸后修复
- 预期差叙事说不清

## 输出语气要求

- 承认这是高风险下注,不要伪装成稳健推荐
- 但也不要只说风险,要把**用户最需要判断的压缩信息**放到最前面:
  - 财报日期
  - 历史反应
  - 预期差来源
  - 入场区间
  - 失效位
  - 财报后第一目标

## 输出要求

最终候选总表、单票展开、报价校验、概率与最终自检统一按 [../output-contract.md](../core/output-contract.md) 执行;财报预判模式额外要求每只显式写出财报日期来源、历史反应统计、预期差叙事和"为什么市场可能还没完全定价"。
